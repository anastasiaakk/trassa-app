# Публикация сайта для других пользователей (свой домен, например `example.com`)

Сайт — это статика из папки **`dist/`** после `npm run build`. Чтобы открывали по адресу вроде **`https://trassa.ru`**, нужны: **домен**, **сервер в интернете** (VPS) и **HTTPS**. Домен **`trassa.ru`** нужно **купить у регистратора** (если свободен) и привязать к серверу — само имя проект не регистрирует.

## Постоянный сайт — надёжный вариант

**Цель:** сайт доступен **круглосуточно**, данные пользователей **не пропадают** после перезапуска.

| Подход | Постоянство | Для этого проекта (Node + **SQLite** на файле) |
|--------|-------------|-----------------------------------------------|
| **VPS** (Timeweb, Selectel, REG.RU Cloud, DigitalOcean, Hetzner и т.д.) | Сервер всегда включён, **диск постоянный** — база лежит в файле как у вас локально | **Рекомендуется:** один VPS, на нём **nginx** + **Node API** + каталог с `app.db` |
| Бесплатные PaaS (Render, Railway…) | Часто «засыпают» или **сбрасывают** файловую систему | Риск **потери SQLite**; нужен платный тариф с **persistent disk** или смена БД |
| Только статика (Netlify/Vercel) + API отдельно | Статика постоянна | Нужно **два** деплоя и согласовать **CORS** и URL API |

**Практический минимум для «навсегда»:**

1. **Оплатить VPS** (часто от ~200–500 ₽/мес в РФ или от ~$4–6 у зарубежных провайдеров) с **Ubuntu**.
2. **Купить домен** (`.ru` / `.рф` и т.д.) у регистратора и в DNS указать **A-запись** на IP VPS.
3. На VPS по чеклисту ниже: **nginx**, **TLS (Let’s Encrypt)**, **Node 22+**, systemd-сервис API, папка с **`dist/`** и с правами на запись для **`server/data/`** (или путь из `TRASSA_DATA_DIR`, если задали).

Дальше — разделы **«Схема»** и **«Краткий чеклист»** (те же шаги, что в **`deploy/nginx-trassa.example.conf`** и **`deploy/trassa-api.service.example`**).

## «Установить бесплатный домен» — что реально можно

**Автоматически зарегистрировать домен за вас нельзя** (ни ассистент, ни репозиторий): нужны **ваш аккаунт**, иногда **оплата** или привязка карты, настройка DNS в панели регистратора.

**Бесплатный постоянный адрес без покупки имени** обычно значит **поддомен хостинга**, например:

- **Netlify**, **Vercel**, **Cloudflare Pages**, **GitHub Pages** — выдают адрес вида **`something.netlify.app`** / **`*.vercel.app`** и HTTPS после деплоя **`dist/`**.
- У **Cloudflare** можно взять **бесплатный поддомен** в зоне `*.pages.dev` при использовании их Pages (условия — на сайте Cloudflare).

У этого проекта есть ещё **backend (Node + SQLite)**. Статику можно выложить на сервис выше; **API** чаще выкладывают **отдельно** (бесплатные уровни у **Render**, **Railway**, **Fly.io** и др.) и в **`CORS_ORIGIN`** на сервере указывают origin фронта (`https://ваш-проект.netlify.app` и т.д.). На бесплатных инстансах **файл SQLite иногда сбрасывается** при перезапуске — для серьёзного использования лучше **VPS** или облачная БД.

**Показать сайт с вашего компьютера на пару дней/для демо:** **Cloudflare Tunnel** или **ngrok** — дают внешнюю ссылку на локальный порт (не замена нормальному хостингу).

## Схема

1. Пользователь открывает **`https://ваш-домен.ru`** — nginx отдаёт файлы из **`dist/`**.
2. Запросы к **`/api/...`** nginx перенаправляет на Node (**порт 4000**) только на **localhost** сервера.
3. Во фронте при сборке: либо **без** `VITE_API_URL` (тогда API на том же домене `/api`), либо с **`VITE_API_URL=https://api.ваш-домен.ru`** (тогда в `CORS_ORIGIN` на API перечислите origin сайта).

Приложение использует **HashRouter** (`/#/…`), поэтому глубокие ссылки не требуют сложной настройки SPA на сервере.

## Краткий чеклист

1. **VPS** (Ubuntu и т.п.), открыты порты **80** и **443**.
2. Установить **Node.js 22+**, **nginx**, при необходимости **certbot** для Let’s Encrypt.
3. Склонировать проект, в корне: `npm ci`, `npm run build`; в **`server/`**: `npm ci`, `npm run build`.
4. На сервере задать **`JWT_SECRET`** (≥32 символов) и переменные из **`deploy/env.api.example`** — см. **`deploy/trassa-api.service.example`**.
5. Скопировать **`dist/`** в каталог веб-сервера (пример путей — в **`deploy/nginx-trassa.example.conf`**).
6. Настроить nginx по примеру, получить сертификат **`certbot --nginx`**, включить **`TRUST_PROXY=1`** в env API (уже в примере).

## Сборка фронта под прод

- Один домен для сайта и API: в **`deploy/env.frontend.build.example`** описан вариант **без** `VITE_API_URL`; затем `npm run build` в корне проекта.
- Отдельный поддомен для API: задайте **`VITE_API_URL=https://api.ваш-домен.ru`** при сборке и добавьте **`https://ваш-домен.ru`** в **`CORS_ORIGIN`** на машине с API.

Файл **`.env.production`** в репозитории заточен под **десктоп-сборку** (Electron). Для сайта удобнее собирать с **`env.frontend.build.example`** через копирование в **`.env.production.local`** или переменные окружения в CI.

## Безопасность

- Не выставляйте Node в интернет без nginx/TLS, если не уверены в настройке.
- **`JWT_SECRET`** только на сервере, длинный и случайный.
- Регулярные обновления ОС и зависимостей.
